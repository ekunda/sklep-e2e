🧠 Teoria · dlaczego Testcontainers

Mocki mogą kłamać

Mock powie „zapis się udał". Prawdziwa baza odrzuci zły typ, brakujący indeks albo złamany UNIQUE.

🧠 Teoria · co warto testować

Transakcja zamówienia — wszystko albo nic

Zamówienie to kilka zapisów naraz: sprawdź stan magazynu, zmniejsz go, zapisz zamówienie. Jeśli cokolwiek po drodze się nie uda → ROLLBACK i magazyn zostaje nietknięty. FOR UPDATE blokuje wiersz, żeby dwa zamówienia nie kupiły tej samej sztuki.

src/routes/orders.routes.ts ↗
await client.query("BEGIN");
for (const item of items) {
  const p = (await client.query(
    `SELECT stock FROM products WHERE id=$1 FOR UPDATE`, [item.productId])).rows[0];
  if (p.stock < item.quantity) {
    await client.query("ROLLBACK");          // za mało → wycofaj CAŁOŚĆ
    return res.status(400).json({ error: "Niewystarczający stock" });
  }
  await client.query(`UPDATE products SET stock = stock - $1 WHERE id=$2`, [item.quantity, p.id]);
}
await client.query("COMMIT");
🔒To dokładnie ten przypadek, gdzie mock by skłamał — Testcontainers sprawdza prawdziwy ROLLBACK i blokady wierszy na żywym Postgresie.
🧠 Teoria · Testcontainers

Prawdziwy kontener prosto z kodu testu

1Test startuje → Testcontainers uruchamia kontener (np. postgres:16).
2Dostajesz gotowy connection string do bazy.
3Test wykonuje operacje na realnej bazie.
4Po wszystkim kontener znika — czysto, bez śmieci.
tests/integration/global-setup.ts ↗
// Start Postgresa + Redisa raz, równolegle — szybciej.
const [pg, redis] = await Promise.all([
  new PostgreSqlContainer("postgres:16").start(),
  new GenericContainer("redis:7").withExposedPorts(6379).start(),
]);
provide("databaseUrl", pg.getConnectionUri());

Podman

Dwie zmienne środowiskowe: DOCKER_HOST na pipe Podmana i TESTCONTAINERS_RYUK_DISABLED=true. Bonus: umie Hyper-V, gdy WSL2 nie działa.

🧠 Teoria · seed per test

Każdy test sieje własne dane

Koniec z globalnym seedem, który psuje się przy każdym nowym teście. Helper zwraca to, czego potrzebują asercje.

tests/helpers/seed.ts ↗
export async function seedUser(pool, options = {}) {
  const opts = { email: `user-${++c}@test.com`, password: "Password123!", ...options };
  const hash = await hashPassword(opts.password);
  const r = await pool.query(`INSERT INTO users ... RETURNING id, email`, [...]);
  return { ...r.rows[0], password: opts.password }; // hasło plain do logowania
}
⚠️Nie zakładaj id = 1. Używaj id zwróconego z seeda — sekwencje nie resetują się między testami (to nas realnie ugryzło w CI).
🧹Izolacja: przed każdym testem czyść bazę — beforeEach(cleanDatabase) robi TRUNCATE … RESTART IDENTITY CASCADE. Dzięki temu test B nie widzi danych testu A.
🧠 Teoria · pułapka Vitest

process.env z global-setup nie dociera do testów

Klasyczny błąd: ustawiasz DATABASE_URL w global-setup, a workery testowe go nie widzą. Rozwiązanie to API provide / inject.

global-setup.ts ↗
// ✅ przekaż przez provide
provide("databaseUrl", uri);
provide("redisUrl", url);
test-app.ts ↗
// ✅ odczytaj przez inject
createApp({
  databaseUrl: inject("databaseUrl"),
  redisUrl: inject("redisUrl"),
});
🧩Dodatkowo: fileParallelism:false — pliki integracyjne dzielą jeden kontener, więc nie mogą czyścić bazy równocześnie.

🧠 Teoria · Playwright fixtures

Fixture = gotowy element wstrzyknięty do testu

page to też fixture. My dokładamy własne: świeży użytkownik i produkt (seed per test) oraz strona z gotowym logowaniem.

tests/e2e/fixtures.ts ↗
export const test = base.extend<Fixtures>({
  testUser: async ({ request }, use) => {
    const res = await request.post("/api/test/seed-user", { data:{ email, password } });
    await use(await res.json());
    await request.delete(`/api/test/cleanup-user/${id}`); // teardown
  },
  authedPage: async ({ page, testUser }, use) => {
    await page.goto("/");
    await page.evaluate(t => localStorage.setItem("auth_token", t), testUser.token);
    await use(page); // logowanie przez API — szybciej niż przez UI
  },
});
🧠 Teoria · pułapka kolejności

Kontenery i serwer startują w global-setup

Playwright najpierw uruchamia wbudowany webServer, a dopiero potem global-setup. Problem: nasz serwer potrzebuje adresów kontenerów, które powstają… właśnie w global-setup. Rozwiązanie: startujemy serwer sami — w global-setup.

1Start kontenerów Postgres + Redis
2Migracje + createApp() z URL-ami kontenerów
3Serwer aplikacji listen(3000) — w tym samym procesie
4Testy klikają po UI; global-teardown sprząta wszystko
🔌Dodatkowo serwer testowy musi dostać NODE_ENV=test — Vitest ustawia to sam, Playwright nie (patrz slajd o bugach z CI).
🧠 Teoria · debugowanie

Trace — czarna skrzynka testu

Przy nieudanej próbie Playwright nagrywa timeline akcji, screenshoty, sieć i konsolę. Otwierasz i widzisz dokładnie, gdzie i kiedy pękło.

  • Czerwona akcja → gdzie pękło
  • Call / Log → dlaczego (timeout vs zła liczba)
  • Network → wina backendu czy UI?
  • Snapshot → stan DOM przed/po akcji
playwright.config.ts ↗
use: {
  trace: "on-first-retry",
  screenshot: "only-on-failure",
  video: "retain-on-failure",
}
🎞️on-first-retry = pełna czarna skrzynka tylko wtedy, gdy trzeba — bez zalewania dysku. Otwierasz: pnpm exec playwright show-trace trace.zip.

Test padł na kliknięciu. W trace, w zakładce Network, widzisz 500 na /api/orders. Kto zawinił?

Złam i napraw 🔧

🧠 Teoria · jak pisać testy

Źle vs dobrze — przykłady

Ten sam cel, dwa podejścia. Po prawej — tak jak w naszych testach z repo.

❌ kruche
// selektor zależny od stylów
page.click(".btn.btn-primary.mt-3");

// sztywny sleep — flaky
await page.waitForTimeout(2000);

// współdzielony email → duplikat
seedUser(pool, { email: "test@test.com" });
✅ stabilne
// selektor po roli / tekście
page.getByRole("button", { name: "Dodaj do koszyka" });

// auto-wait — czeka sam
await expect(toast).toHaveText("Dodano…");

// unikalny email + czysta baza
await seedUser(pool);
🎯Dobry test jest deterministyczny: ten sam wynik niezależnie od kolejności i prędkości maszyny. Więcej par w szkolenie/PORADNIK-PROWADZACEGO.md.
🧠 Teoria · higiena testów

Anti-flaky checklist

  • Używaj await expect() — auto-wait, nie raw assertions
  • Nigdy waitForTimeout() w teście
  • Czyść bazę w beforeEach
  • Generuj unikalne dane (timestamp/random)
  • Mockuj zewnętrzne API w E2E
  • Czekaj na response API przed asercją UI
  • Realistyczne timeouty — nie za krótkie
  • --repeat-each=10 → poluj na flaki
🎯Lepiej mieć 5 stabilnych testów niż 50 flaky.
